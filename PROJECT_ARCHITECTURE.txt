KEY COMPONENTS AND INTERACTIONS
================================================================================

1. ENTRY FLOW (main.py)
   
   User starts main.py → initializes NautilusManager and DiscoveryScript
   │
   ├─ Discovery Phase (user not yet identified)
   │  ├─ DiscoveryScript.process_initial_response() reads input
   │  ├─ Extracts machine and skill level
   │  ├─ Locks manager.machine_id, manager.mode, manager.skill_declared
   │  └─ Returns confirmation: "Session Locked. Mode: PRO. Machine: ..."
   │
   └─ Manager Phase (user identified)
      ├─ NautilusManager.ask() reads input
      ├─ Checks safety triggers (mode rules first, then global)
      │  ├─ If matched → returns safety message (interrupt_triggers hit)
      │  └─ If not matched → returns STF diagnostic response
      └─ Repeats until user exits

2. KNOWLEDGE BASE

   NautilusCore (nautilus_core.py):
     - On init: loads data/machine_library.json
     - Provides get_info(machine_id) for manager to look up machine-specific data
     - Used by manager to enrich responses (optional lookups)

3. RULES SYSTEM

   YAML File Structure:
     - global.yaml: cross-cutting policies (STF philosophy, identity gate, etc.)
     - beginner.yaml: beginner-mode era logic and safety
     - pro.yaml: pro-mode era logic and safety

   Rule Keys:
     - era_logic: era-specific diagnostic paths
     - safety_logic: interrupt_triggers + (message or warning_message)
     - response_style: tone and detail level

   Manager Rule Loading (in _load_rules):
     - Loads global.yaml first
     - Loads mode-specific YAML (beginner.yaml or pro.yaml)
     - Returns {"global": global_rules, "mode": mode_rules}

   Manager Safety Lookup (in ask):
     - Checks mode_rules['safety_logic'] first
     - Falls back to global_rules['safety_logic'] if not found in mode
     - Accepts either 'message' or 'warning_message' key
     - Looks for 'interrupt_triggers' key (list of keywords)
     - If any trigger matches user_input (case-insensitive), returns alert message

4. SESSION STATE

   NautilusManager attributes:
     - machine_id (set by DiscoveryScript)
     - skill_declared (set by DiscoveryScript)
     - mode (set by DiscoveryScript, maps skill → beginner, pro, etc.)
     - rules (loaded on demand, None until skill_declared is True)
     - librarian (NautilusCore instance for data lookups)

5. RESPONSE FORMATTING (STF)

   When user is identified and no safety trigger:
     - Prefix: "[PD MODE_NAME MODE]"
     - Body structure:
       * [STRAIGHT]: physical/mechanical checks
       * [TRUE]: electrical/measurement validation
       * [FLUSH]: root-cause identification and fix

DIAGNOSTIC CHEAT SHEET INTEGRATION
================================================================================

Machine Library Structure (data/machine_library.json):

The machine library now contains comprehensive diagnostic data across six eras:
  1. EM (Electromechanical, 1960s-1970s)
  2. Solid State Early (Gottlieb/Bally 1970s-1980s)
  3. Bally MPU (1978-1993 Bally systems with LED diagnostics)
  4. Solid State Modern (Williams WPC 1989-1999)
  5. Digital (Stern 2000+)

Each machine entry includes:
  - id: Unique machine identifier (e.g., "BALLY_1978_MPU")
  - name: Human-readable machine name
  - era: Machine era classification
  - manufacturer: Manufacturer name
  - symptoms: Array of diagnostic symptom objects

Each Symptom Object includes:
  - symptom: Description of the fault/issue
  - diagnosis: STF-formatted diagnostic path [STRAIGHT/TRUE/FLUSH]
    * STRAIGHT: Visual/mechanical inspection steps
    * TRUE: Electrical/measurement validation steps
    * FLUSH: Root cause identification and repair steps
  - common_causes: Array of typical failure modes
  - tools_needed: Required tools or replacement parts
  - led_code: (Bally MPU only) LED flash pattern for identification

Examples:

EM Common Unit Reset Symptoms:
  - Unit not resetting after game
  - Playfield lamps not lighting
  - Slingshot not firing

Bally 1978 MPU LED Flash Codes:
  - "1 flash (0.5s pause)" → ROM checksum failure
  - "2 flashes (0.5s pause pattern)" → RAM failure
  - "4+ flashes rapid" → Solenoid driver short or clock failure
  - "No flash/dark" → Power supply failure

Williams WPC 90s Symptoms:
  - Fuse blowing on startup (short circuit diagnosis)
  - Bumpers not firing (solenoid/driver check)
  - Display corrupted or missing segments

Retrieval During Session:

NautilusManager can call NautilusCore methods to enrich diagnostics:
  1. By Era: core.get_symptoms_by_era("EM")
     → Returns all EM machine symptoms for beginner prompts
  
  2. By Machine: core.get_symptoms_by_machine("BALLY_1978_MPU")
     → Returns all symptoms for specific identified machine
  
  3. By Keyword: core.search_symptom_by_keyword("solenoid")
     → Helps match user-described issues to known symptoms
  
  4. LED Codes: core.get_led_flash_codes()
     → Provides Bally MPU LED diagnostic reference during troubleshooting

Database Statistics:
  - 6 machine eras defined
  - 20+ individual symptoms with full diagnostic paths
  - 5 Bally MPU LED flash code references
  - Cross-indexed by era, machine, and keyword

CONCLUSION
================================================================================

Project Nautilus is a compact, rule-driven diagnostic engine for pinball
machines. The architecture separates concerns into:
  - Rules (YAML in rules/)
  - Logic (Python in logic/)
  - Data (JSON in data/)
  - CLI interface (main.py)

The STF philosophy (Straight/True/Flush) provides a consistent diagnostic
framework. Safety interrupts are pulled dynamically from YAML, enabling
quick updates without code changes. The system has been validated for
syntax, imports, and basic runtime behavior.

Future enhancements should focus on unit tests, error handling, and
potentially more sophisticated NLP for discovery parsing.
